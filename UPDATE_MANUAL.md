# TypeNo Agent 更新手册

本文件用于说明 `TypeNo Agent` 当前版本的真实产品定义、历史演进、维护边界和后续优化方向。  
后续接手开发时，应优先以本文“当前版本基线”和 `CHANGELOG.md` 的最新版本记录为准，不要把早期阶段口径误当成现状。

## 当前版本基线

- 基础项目：`marswaveai/TypeNo`
- 当前工作副本：`TypeNo Agent`
- 当前 app 名称：`TypeNo Agent`
- 当前 bundle id：`ai.marswave.typeno.agent`
- 当前安装位置：`~/Applications/TypeNo Agent.app`
- 当前阶段基线：`1.2.6` 这条多模式版本线

## 先看这一条：哪些说法已经过时

以下说法属于早期阶段口径，现已淘汰：

- “当前只保留两个模式：普通模式 / LLM Agent 模式”
- “这个 fork 的目标只是把原版 TypeNo 改成一个双模式 Agent 输入工具”

这些说法对应的是 `1.1.0` 阶段。  
从 `1.2.6` 开始，`TypeNo Agent` 已经进入多模式版本线，`口语整理` 和其他风格模式都属于现行功能，而不是历史遗留分支。

## 当前项目定位

`TypeNo Agent` 不是原版 `TypeNo` 的简单镜像，也不是只做“语音转文字”的极简工具。  
它的当前定位是：

- 以本地语音转写为底座
- 以菜单栏常驻和快捷键触发为交互主轴
- 在原始转写后，按模式决定是否进行 LLM 重写
- 面向真实工作流提供不同输出风格

它现在更接近“多模式语音改写与 Agent 输入工具”，而不是“原版 TypeNo 的最小改版”。

## 当前已完成改造

### 1. 独立 app 身份

- 已从原版 `TypeNo` 中拆分为独立 app
- 已修改 app 名称、可执行名、bundle id 和打包产物名
- 目的：避免与原版共用系统权限记录，便于并存测试和长期维护

### 2. 多模式体系

当前可用模式如下：

- `普通`
- `Agent`
- `口语整理`
- `中英夹杂`
- `日漫中二`
- `网络热梗`
- `电影台词风`
- `哲学社会学黑话`
- `阴阳吐槽`

说明：

- `普通` 是原始转写后直接输出
- `Agent` 是任务型结构化重写
- `口语整理` 是“只整理，不换风格”的轻量重写
- 其余模式属于明确保留的风格化输出模式

### 3. 默认快捷键映射

- 左 `Option`：默认模式
- 左 `Control`：`口语整理`
- 右 `Control`：`Agent`

### 4. LLM Provider 接入

- 已支持 provider 切换：
- `Qwen`
- `Kimi`

- 当前实现使用 OpenAI-compatible `chat/completions` 接口
- 当前默认模型：
- `Qwen` -> `qwen-plus`
- `Kimi` -> `moonshot-v1-8k`

### 5. API Key 管理

- 不再依赖终端环境变量作为唯一入口
- API key 按 provider 分开保存到 Keychain
- 菜单中可直接设置当前 provider 的 key

### 6. Agent 模式输出结构

当前 `Agent` 模式输出被强制收敛为固定三段式骨架：

```text
任务：
检查项： 或 要求：
输出：
```

说明：

- 第一段固定为 `任务：`
- 第二段会根据意图在 `检查项：` 和 `要求：` 之间选择
- 第三段固定为 `输出：`

目标：

- 让输出更适合直接喂给 agent
- 减少模型自由发挥
- 避免官样话和空泛表达
- 尽量保留用户原始语言和真实工作目标

### 7. 输出风格收敛

当前已经明确加入这些约束：

- 保留原语言
- 不擅自翻译成英文
- 不擅自改造成别的任务
- 对中文口语请求，优先输出中文任务单
- 尽量短、实用、可执行
- `口语整理` 优先“删噪音”，而不是“重写成另一种文体”

## 当前使用方式

### 普通模式

1. 打开 `TypeNo Agent`
2. 确认默认模式为 `普通`
3. 直接录音
4. 应用会本地转写并自动粘贴

### Agent 模式

1. 打开 `TypeNo Agent`
2. 在菜单中选择 provider：`Qwen` 或 `Kimi`
3. 在菜单中设置该 provider 的 API key
4. 将默认模式切为 `Agent`，或直接使用 `Agent` 快捷键
5. 正常录音
6. 应用会先本地转写，再用 LLM 重写为结构化任务单，最后自动粘贴

### 其他风格模式

- 先完成本地转写
- 再根据当前模式走对应的 LLM 重写 prompt
- 最终结果仍按原流程自动粘贴或复制

## 当前已知情况

### 1. 本地语音模型首次下载

- `coli` 第一次运行时会下载一个大约 `155MB` 的语音模型
- 模型未下载完成前，原版和改版都可能卡在转写阶段
- 这个问题已经确认，不属于 `TypeNo Agent` 模式改造本身的 bug

### 2. Provider 可用性

- `Qwen` / `Kimi` 的接口和模型权限与账号状态相关
- 如果被 provider 拒绝，优先检查：
- API key 是否正确
- 当前账号是否有该模型权限
- 当前 provider 的模型名是否可用

### 3. 当前优化重心

- 当前已经不是“功能不可用”阶段
- 当前主要问题是：
- 模式体验是否稳定
- prompt 是否跑偏
- 菜单和状态提示是否足够清楚
- 文档和代码口径是否一致

## 后续如何跟进 upstream 更新

建议采用“人工挑选合并”的方式，而不是直接自动覆盖。

当前约定如下：

- 菜单中的“检查上游更新”只用于查看 upstream `TypeNo` 的新版本
- upstream 更新代表基础能力来源，不代表可直接安装到 `TypeNo Agent`
- `TypeNo Agent` 必须保留自己的定制层，不能被 upstream app 包直接覆盖
- 正确做法是：先更新 upstream 源码，再把 `TypeNo Agent` 的定制内容原封不动地重新合并回来

### 推荐流程

1. 先查看 upstream 新版本 release note
2. 拉取或对照 upstream 源码变更，而不是直接安装 upstream 发布包
3. 先把 upstream 的核心源码更新到当前工作副本基线
4. 再把 `TypeNo Agent` 的定制内容按原意重新合并回来，包括但不限于：
- app 身份和打包产物
- 中文化文案
- provider 切换
- Keychain API key 管理
- `Agent` 模式 prompt 结构
- `口语整理` 与其他风格模式
- 快捷键和菜单层定制
5. 合并前先判断是否会发生冲突，重点检查：
- 稳定性修复
- 热键/录音体验优化
- UI 文案或小交互优化
- 更新机制改动
- 与本 fork 无关的功能
6. 对有冲突的部分，先做差异分析，再决定保留 upstream、保留 fork，还是做手工整合
7. 优先合入这些 upstream 内容：
- 热键相关改进
- 录音/转写稳定性修复
- 超时、代理、模型下载体验修复
- 菜单栏和状态提示优化

8. 谨慎处理这些内容：
- 自动更新逻辑
- app 身份相关改动
- 会覆盖本 fork 的菜单结构、模式结构和 provider 逻辑的改动

9. 合并后必须回归测试：
- 普通模式录音
- Agent 模式录音
- 口语整理模式录音
- Qwen / Kimi provider
- 默认快捷键
- Agent 快捷键
- 中文界面主流程

### 不建议的方式

- 不建议每次 upstream 更新后直接整包覆盖
- 不建议把 upstream 发布的 `TypeNo.app` 直接覆盖安装到 `TypeNo Agent.app`
- 不建议把 upstream release 资产当成 `TypeNo Agent` 自身更新包使用
- 不建议在未测试的情况下同步自动更新逻辑

### 当前维护策略

当前这条线适合长期采用：

- upstream 提供基础能力和稳定性更新
- `TypeNo Agent` 保留自己的定制层：
- 独立 app 身份
- 多模式体系
- provider 切换
- API key 管理
- 中文界面
- `Agent` 结构化输出
- 风格模式调校

## 交接用架构分层

本节用于跨人、跨项目、跨模型、跨对话交接。  
后续任何人接手 `TypeNo Agent` 时，都应先用这一节判断：哪些内容跟随 upstream，哪些内容必须保留为 fork 定制，哪些内容不能直接自动合并。

先强调一条总原则：

- `源码层 / 定制层 / 高冲突层` 这套框架本身是长期有效的
- 但具体某段代码在某个时点归入哪一层，不是永远固定的
- upstream 会更新，fork 也会继续更新，所以每次合并前都必须基于最新 diff 重新判断一次

换句话说：

- 分层框架是稳定的
- 具体归类是动态的

典型情况：

- 某块代码原本看起来像 fork 定制
- 但 upstream 后来在同一区域也加了新逻辑
- 那它就不再适合继续视为“纯定制层”，而应升级为“高冲突层”

反过来：

- 某块代码以前因为双方都改过，所以算高冲突层
- 如果后来 upstream 不再改，而 fork 也只是在外围调用它
- 那它也可能重新回落为源码层或定制层

### 一、源码层

源码层指 upstream `TypeNo` 的基础能力层。  
这部分原则上优先跟随 upstream 演进，除非明确证明 upstream 改动会破坏 `TypeNo Agent` 定制层。

当前应归入源码层的内容：

- 菜单栏常驻 app 与基本启动流程
- 麦克风 / 辅助功能权限检测与打开系统设置
- 本地录音能力
- 基于 `coli` 的本地语音转写
- 转写后复制到剪贴板并粘贴回前台应用
- 文件转录入口
- 录音计时器、长录音动态超时等基础体验能力
- overlay / 状态提示的基础交互骨架

可以把源码层理解为：

- `录音 -> 本地转写 -> 粘贴`

### 二、定制层

定制层指 `TypeNo Agent` 在 upstream 之上新增或重写的产品能力。  
这部分不能被 upstream 直接覆盖，合并时必须原封不动地保留或人工回贴。

当前应归入定制层的内容：

- `TypeNo Agent` 的 app 身份：
- app 名称
- 可执行名
- bundle id
- 打包产物名

- 中文化与本地化定制：
- 中文菜单文案
- 中文状态提示
- 当前文档体系与项目说明

- 大模型接入层：
- `Qwen` / `Kimi` provider
- OpenAI-compatible `chat/completions` 调用
- Keychain 分 provider 存储 API key

- 重写能力层：
- `Agent` 模式
- `口语整理`
- `中英夹杂`
- `日漫中二`
- `网络热梗`
- `电影台词风`
- `哲学社会学黑话`
- `阴阳吐槽`

- `Agent` 模式的结构化 prompt 规则：
- `任务：`
- `检查项：` 或 `要求：`
- `输出：`

- 当前模式绑定的默认快捷键设计：
- 左 `Option` = 当前默认模式
- 左 `Control` = `口语整理`
- 右 `Control` = `Agent`

- 上游更新跟踪策略：
- 只检查 upstream 更新
- 不直接安装 upstream app 包
- 先升级源码，再回贴定制层

可以把定制层理解为：

- `TypeNo Agent = TypeNo 核心能力 + LLM 改写产品层`

### 三、高冲突层

高冲突层指 upstream 和 fork 都容易同时改动的部分。  
这些内容不能简单按“保留 upstream”或“保留 fork”处理，必须逐项分析后手工合并。

当前高冲突层主要包括：

- 热键系统
- 菜单栏 UI 与菜单结构
- 状态提示与 overlay 文案
- `AppState` 主流程
- 更新检查与更新提示逻辑

原因：

- upstream 可能在这些地方持续优化基础体验
- fork 也在这些地方绑定了模式、provider、中文化和产品语义

### 四、快速判断规则

后续有人接手时，可以先按下面规则判断归属：

- 如果一段改动只影响“录音、权限、本地 ASR、粘贴、基础状态机”，先按源码层处理
- 如果一段改动只影响“LLM、provider、Keychain、多模式、中文化、TypeNo Agent 身份”，先按定制层处理
- 如果一段改动同时影响“菜单、热键、状态提示、更新逻辑、主流程状态机”，先按高冲突层处理

补充规则：

- 不要因为某段代码“上次被归到源码层”就默认这次仍然是源码层
- 不要因为某段代码“最初是 fork 自己写的”就默认它永远属于定制层
- 每次遇到 upstream release 或 fork 新改动时，都要按最新 diff 动态重判

### 五、合并 upstream 的标准流程

后续每次跟进 upstream 时，统一按这个顺序执行：

1. 查看 upstream release note 和源码 diff
2. 查看当前 fork 在同一时期新增或修改了哪些内容
3. 先按最新 diff 重新判断：哪些属于源码层，哪些属于定制层，哪些升级为高冲突层
4. 先升级 upstream 源码层
5. 再把 `TypeNo Agent` 定制层原封不动地回贴
6. 对高冲突层逐项比较差异，手工整合
7. 完成后回归测试核心流程

### 六、交接时必须先确认的六块定制能力

无论是换人、换模型还是换对话，开始工作前都先确认下面六块有没有被冲掉：

- app 身份层
- provider 层
- Keychain 层
- LLM rewrite 层
- 多模式层
- 中文化与菜单层

如果这六块有任何一块不完整，就不要直接继续开发，先修复基线再往下做。

3. 如果文件属于高冲突文件，再进入文件内部按“源码层 / 定制层 / 高冲突层”继续拆

### 八、`main.swift` 内部的二次拆分规则

因为当前仓库的核心代码几乎都集中在 `Sources/Typeno/main.swift`，所以合并 upstream 时不能只停在“文件级”。  
进入这个文件后，建议按下面顺序继续拆：

1. 先找源码层主链路：
- 启动流程
- 权限
- 录音
- `coli` 转写
- 粘贴

2. 再找定制层插入点：
- provider
- Keychain
- LLM rewrite
- 多模式
- 中文化菜单与文案
- `TypeNo Agent` 身份相关语义

3. 最后找高冲突段：
- 热键体系
- 状态栏菜单
- `AppState`
